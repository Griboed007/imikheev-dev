~/work/geofence-twin

Semantic digital twin for geofence anomaly detection

role architect · builder context Nebius Serverless AI Challenge when jun 2026 status shipped writing the article ↗

Overview

Fleet telemetry tells you where a vehicle is. It rarely tells you whether that's normal. This project builds a semantic digital twin over GPS trajectory data: a hybrid model predicts where a vehicle should be next and scores deviations, while a SAREF/RDF knowledge graph makes every observation, zone, and anomaly queryable in SPARQL.

The bet: anomaly detection gets dramatically more useful when its outputs live in a knowledge graph instead of a log file — because then the question "which vehicles crossed which zones abnormally, and when" is one query, not a data-engineering project.

The system

A batch job enriches raw trajectories into typed observations and writes them as RDF triples; the endpoint serves the hybrid prediction/scoring model. Everything runs as containers on Nebius serverless — the whole pipeline is reproducible from a clean account.

~/methods/spec-driven-agents

Spec-driven agentic development

source framework.md · distilled from deckgen stack openspec · claude code · plan mode status v1 · in daily use

TL;DR

the whole framework in four lines

Write the contract first (spec), prove the behavior first (tests), and prove it for real last (independent read-only verification plus a live run). Keep schema and types welded through migration files. Split the work across a planner, an implementer, and a human who approves the irreversible. Never mutate state out-of-band.

A method for building software with AI agents where intent, schema, and code never drift apart, nothing is "done" until it runs live, and state is never mutated out-of-band. Three disciplines stacked: spec-driven, test-driven, verification-driven.

Three roles, deliberately separated

rolewhoownsnever does
strategistplanning model in chatproposals, scoping, review, independent verificationwrites the implementation
implementerclaude codetest-first implementation, the repo, migrations through the repo workflowapplies schema out-of-band; skips tests
operatorthe humanapproving plans, running the live system, the live definition of donerubber-stamps without the live run

The separation is the point. Because the Strategist did not write the code, it stays adversarial to the implementer's output — it catches the class of bugs that self-review misses: a policy that passes its own test but has a logic hole, a value that's internally consistent but wrong against the source.

The loop

Every capability starts as a written change proposal before any code. The change isn't finished until a human has clicked through the live seams — and then it's folded back into the durable spec, so the spec always reflects shipped reality, not a stale plan.

The proposal anatomy

Every proposal has the same shape: Why (the problem and the thesis of the fix), What changes, Impact — including an explicit statement of what is NOT touched, because naming the blast radius is half the value — Design with alternatives considered and the honest hard parts called out, a Spec delta written as WHEN/THEN scenarios (behavioral and testable, no implementation detail), Tasks in TDD order ending in a manual live-DoD group, and Notes for the implementer with hard fences like: "This change adds no DDL. If a task seems to need schema, stop — it's out of scope."

If a requirement can't be phrased as an observable scenario, it isn't a requirement yet.

Independent verification — trust, but verify

After the implementer reports done, the Strategist verifies read-only: query real data to confirm the invariants actually hold in live rows, read the actual function and policy bodies to confirm the hole isn't there, run advisors after any schema change, and believe the live system over the test report.

from the deckgen run

Every figure on a generated deck resolving to a ledger row — 56/56, zero dangling — checked with SQL, repeatedly, at scale. And an invitation-accept RPC read line-by-line to confirm it rejected a caller whose identity didn't match the invited address: the crux that closed an "any token = any project" hole. Tests prove the happy path; reading the code proves the hole isn't there.

The live run is the real definition of done

Headless tests go green and still miss the seams — the wiring between components: the upload that should trigger extraction, the button that should start a job, the regeneration that only runs through a real task runner. So every change carries two tracks: Track A, headless tests — necessary, not sufficient. Track B, the live run — this is the gate. A change with green tests and no live run is not done; it's untested where bugs actually live.

CLAUDE.md as accumulated memory

Per-repo and per-app CLAUDE.md files hold conventions and hard-won, non-obvious lessons. Hit a footgun that wasn't predictable from the docs? Encode it so it's never re-learned — like the rule that stopped a recurring structured-output bug: never put an unconstrained array or a critical nullable field in that model's schema, always cap output tokens.

Bootstrapping a new project

01initialize the spec system (openspec) at the repo root
02write claude.md — root conventions plus per-app stubs
03wire mcp servers: read-only db, playwright, live docs — and turn on plan mode
04stand up test harnesses per layer before the first feature
05establish the migration workflow and least-privilege key split on day one
06run every capability through the loop
07keep the deferred-items ledger from the first session
